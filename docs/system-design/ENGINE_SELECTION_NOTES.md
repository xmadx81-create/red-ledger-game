# 게임 엔진/프로토타입 방식 선택 노트

이 문서는 `red-ledger-game`의 1차 MVP를 어떤 방식으로 구현할지 판단하기 위한 엔진/프로토타입 선택 기준을 정리합니다.

## 목적

`적혈의 장부`는 전투 액션 중심 게임이 아니라, 다음 요소가 핵심입니다.

```text
문서형 UI
자원 관리
선택지 처리
Day 진행
캐릭터 반응
엔딩 플래그
행정 누아르 분위기
```

따라서 첫 MVP에서는 그래픽 성능보다 다음이 중요합니다.

```text
빠른 구현
데이터 연결 용이성
UI 수정 용이성
선택지/이벤트 구조 구현 편의성
모바일 화면 대응
향후 확장 가능성
```

## 후보 방식

MVP 기준 후보는 다음과 같습니다.

```text
1. 웹 기반 프로토타입
2. Godot UI 중심 프로토타입
3. Unity UI Toolkit 또는 uGUI 기반 프로토타입
4. Ren'Py 기반 비주얼노벨+운영 시뮬레이션 확장
```

## 비교표

| 후보 | 장점 | 단점 | 적합도 |
|---|---|---|---|
| 웹 기반 프로토타입 | 빠른 UI 구현, Google Sheets/JSON 연결 쉬움, 모바일 확인 쉬움 | 게임 앱 패키징은 후속 작업 필요 | 매우 높음 |
| Godot | 가볍고 게임 구조화에 좋음, UI 게임 구현 가능, 오픈소스 | 데이터/웹 연동은 직접 구성 필요 | 높음 |
| Unity | 상용 확장성, 모바일 배포 강함, 에셋 생태계 큼 | 초기 MVP에는 무겁고 UI 반복 수정 비용이 큼 | 중간 |
| Ren'Py | 텍스트/선택지/스토리 구현이 빠름 | 경영 시뮬레이션 UI와 데이터 확장에는 한계 | 중간 |

## 이 프로젝트의 핵심 요구

### 1. 데이터 중심 구조

현재 기획은 이미 Google Sheets 기반으로 정리되어 있습니다.

```text
03_자원
04_캐릭터
05_이벤트
06_선택지
07_밸런싱
08_UI화면목록
```

따라서 MVP는 이 데이터를 JSON 또는 내부 객체로 불러와서 화면과 연결하는 구조가 적합합니다.

### 2. 문서형 UI

게임의 분위기는 화려한 전투보다 문서, 장부, 봉인 보고서, 재고표, 선택 결과에 의해 형성됩니다.

따라서 UI를 빠르게 수정하고 테스트할 수 있는 방식이 중요합니다.

### 3. 선택지와 결과 처리

MVP의 핵심은 다음입니다.

```text
이벤트 표시
선택지 표시
자원 변화
인물 반응
엔딩 플래그 누적
Day 진행
```

이 구조는 복잡한 3D 엔진보다 데이터 기반 UI 앱 구조와 더 잘 맞습니다.

## 추천 결론

### 1차 추천: 웹 기반 프로토타입

1차 MVP는 웹 기반 프로토타입을 추천합니다.

이유:

- Google Sheets 데이터를 JSON/CSV로 옮기기 쉽습니다.
- Figma 화면 구조와 맞추기 쉽습니다.
- 모바일 화면 테스트가 빠릅니다.
- 선택지/이벤트/자원 변화 구조를 가장 빠르게 검증할 수 있습니다.
- GitHub와 연결해 버전 관리하기 쉽습니다.

웹 MVP는 다음 방식으로 구성할 수 있습니다.

```text
HTML/CSS/JavaScript
또는
React 기반 단일 페이지 앱
```

초기에는 복잡한 프레임워크보다 단순한 구조가 좋습니다.

## 2차 추천: Godot

게임 앱 형태와 향후 배포까지 생각하면 Godot도 좋은 후보입니다.

적합한 이유:

- UI 중심 게임 구현 가능
- 가벼운 2D 게임에 적합
- 오픈소스 기반이라 시작 부담이 낮음
- 나중에 데스크톱/모바일 앱으로 확장 가능

다만 초기 기획 검증 속도는 웹이 더 빠릅니다.

## 3차 후보: Unity

Unity는 장기적으로 확장성이 높지만, 현재 단계에서는 과할 수 있습니다.

Unity가 적합해지는 시점:

- 캐릭터 애니메이션을 본격화할 때
- 모바일 상용 배포를 강하게 고려할 때
- 에셋 기반 UI/연출이 많아질 때
- 장기 운영형 게임으로 확장할 때

현재 MVP 단계에서는 우선순위가 높지 않습니다.

## 4차 후보: Ren'Py

Ren'Py는 텍스트와 선택지 중심에는 강하지만, `적혈의 장부`의 자원 관리, 데이터 테이블, 운영 화면, 밸런싱 구조를 생각하면 보조 후보입니다.

적합한 경우:

- 스토리 체험판을 빠르게 만들 때
- 캐릭터 대화와 루트 중심으로 검증할 때
- 운영 시스템보다 분위기와 서사 반응을 먼저 테스트할 때

## 최종 권장 전략

```text
1단계: 웹 기반 MVP로 핵심 루프 검증
2단계: 반응이 좋으면 Godot 또는 웹앱 고도화 검토
3단계: 캐릭터 연출/상용 배포가 필요하면 Unity 검토
```

## 웹 MVP 장점

- Figma와 화면 구조를 빠르게 맞출 수 있습니다.
- Google Sheets 데이터를 CSV/JSON으로 변환해 넣기 쉽습니다.
- GitHub Pages 같은 방식으로 테스트 공유가 가능합니다.
- 모바일 브라우저에서 바로 확인할 수 있습니다.
- 기획 수정이 빠르게 반영됩니다.

## 웹 MVP 단점

- 완성 게임 앱처럼 보이려면 후속 패키징이 필요합니다.
- 네이티브 모바일 기능은 제한적입니다.
- 장기적으로 고급 연출/사운드/저장 시스템은 추가 설계가 필요합니다.

## Godot 전환 기준

다음 조건이 생기면 Godot 전환을 검토합니다.

```text
모바일 앱 형태로 빌드하고 싶다.
UI 전환 연출을 게임 엔진 안에서 관리하고 싶다.
캐릭터/배경/애니메이션을 본격적으로 넣고 싶다.
웹보다 게임 파일 구조가 더 필요해진다.
```

## Unity 전환 기준

다음 조건이 생기면 Unity 전환을 검토합니다.

```text
상용 모바일 게임 수준의 연출이 필요하다.
캐릭터 수집/성장/애니메이션이 핵심이 된다.
외부 에셋과 플러그인을 적극 활용한다.
장기 라이브 운영까지 고려한다.
```

## Ren'Py 사용 기준

다음 경우 Ren'Py를 별도 스토리 프로토타입용으로 사용할 수 있습니다.

```text
캐릭터 대화 루트 테스트
프롤로그/개인 서사 테스트
분기형 텍스트 체험판 제작
```
